# Demo Script

This script is designed for presenting the project in class, review meetings, or recorded demos.
It avoids risky live changes and focuses on stable user flows.

## Demo Goals

Show that the system can:

- Load the deployed frontend.
- Authenticate users.
- Manage profile information.
- Create a survey.
- Generate an invite code.
- Fill a survey from the invite code.
- Show survey-related records.
- Explain Render deployment.
- Explain backend API health.

## Preparation

Before the demo:

1. Confirm frontend is live.
2. Confirm backend is live.
3. Confirm test account credentials.
4. Confirm database is connected.
5. Create a backup invite code if needed.
6. Open Render dashboard in another tab.
7. Open GitHub repository in another tab.
8. Keep this script open.

## Common Demo Questions

### Why separate frontend and backend?

Answer:

```txt
The frontend is optimized for static hosting, while the backend needs a Python runtime and database access.
Keeping them separate makes deployment and scaling clearer.
```

### Why does survey fill not require login?

Answer:

```txt
Respondents only need an invite code.
This lowers friction and matches common survey workflows.
The creator still needs login to create and manage surveys.
```

### How is expiration handled?

Answer:

```txt
Each survey has a deadline.
The backend checks the deadline before returning fill content and before accepting responses.
Expired surveys return a specific status so the frontend can show an expired message.
```

### What happens if the frontend calls the wrong API URL?

Answer:

```txt
The static frontend site may return HTML for API paths.
The frontend helper avoids this by using the backend Render URL in production.
```

### What data is stored for responses?

Answer:

```txt
Responses are stored with the survey template ID and answer JSON.
Identified surveys also include respondent identity.
Anonymous surveys can leave identity empty.
```
